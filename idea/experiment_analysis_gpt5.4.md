> 分析脚本口径：以 `records_*.jsonl` 去重（按 `subtask_id` 保留最后一条）后的 278 个 subtask 为准。

## 2. 失败模式分析（125 个失败 subtask）

按导航步数把失败分为三类（成功阈值 `success_distance = 0.25m`，非常严格）：

| 失败模式 | 数量 | 占失败比 | 平均最终距目标 | 解读 |
| --- | --- | --- | --- | --- |
| **A. 提前放弃**（<10 步） | 62 | **50%** | 5.4m | 过早判定"已完成/无目标"就停止 |
| B. 中途失败（10–49 步） | 33 | 26% | 6.0m | 探索方向错误，越走越偏 |
| C. 探索超时（≥50 步） | 30 | 24% | 7.3m | 目标远/大场景，探索没覆盖到 |

### 2.1 核心问题：一半失败是"提前放弃"

**50% 的失败在 10 步内就停止**，且平均离目标还有 5.4m。这指向 **闭环判定（task check / 停止决策）过于激进** —— VLM 过早给出"找到了"或"无此目标"的判断。

更关键的证据 —— **"差一点就成功"的案例**：

- 有 **19 个失败** 是"<10 步就停 + 最终距离 < 2m"
- 其中多个距离在 **0.25–0.5m**，仅比阈值差一点点：

| subtask | 类型 | 目标 | 最终距离 | 步数 |
| --- | --- | --- | --- | --- |
| 00813-svBbv1Pavdk_0_7 | image | rug | **0.27m** | 5 |
| 00880-Nfvxx8J5NCo_0_1 | desc | mirror | **0.28m** | 1 |
| 00824-Dd4bFSTQ8gi_0_4 | object | flower vase | 0.34m | 5 |
| 00810-CrMo8WxCyVb_0_4 | image | refrigerator | 0.44m | 2 |
| 00810-CrMo8WxCyVb_0_6/0_7 | desc/image | refrigerator | 0.44m | 1–3 |

→ 这些几乎是"站在目标旁边但判定停止/未对准 viewpoint"。**0.25m 阈值极严**：成功案例的最终距离中位数才 0.113m，最大 0.246m（贴着阈值）。说明 **最后一步的精细对准（viewpoint alignment）是成功率的隐形瓶颈**。

> ⚠️ **本节结论已被 §2.3 的轨迹级分析修正**："<10步就停"绝大多数并非"提前放弃"，而是 VLM 自信但错误地判定"已到达"（假阳性）。步数分桶只是表象，§2.3 给出了更准确、更可操作的归因。

### 2.2 探索类失败（A+C 尾部）

- 探索超时（≥50 步）平均离目标 7.3m，说明 **frontier 探索在大 scene 里效率不足**，绕远路或反复横跳。
- 这也反映在 SPL 偏低（33.65%）：即便成功，路径也远非最优。

### 2.3 轨迹级二次分析（2026-07-05）：重新定性"提前放弃" ★核心修正

**方法**：不再按步数分桶，而是读取 `trajectory_*.jsonl` 里每一步的 `end_check` 事件 + `goal_obj_ids_mapping`（GT 物体 id ↔ agent 自己场景图检测 id 的映射），判断"停止那一刻到底发生了什么"：VLM 是否喊过 `end_check_yes`；如果喊过，agent 当时锁定的物体是否真的是 GT 对应的检测实例。125 个失败重新分三类：

| 类别 | n | 占比 | 最终距离 均值/中位 | 含义 |
| --- | --- | --- | --- | --- |
| **A. 锁对物体，停远了** | 36 | 29% | 2.52m / 1.39m | `end_check` 判"yes"时锁定的确是 GT 对应实例，但停的位置离标注 viewpoint 还很远 |
| **B. 锁错了物体** | 50 | 40% | 7.84m / 6.53m | `end_check` 判"yes"，但锁定物体根本不是 GT 实例——**从源头就认错了目标** |
| **C. 从未判"yes"** | 39 | 31% | — | 真正意义上没能确认到达 |

**A+B 共 86 个（占全部失败 69%），平均步数仅 10.6～10.7 步** —— 这正是 §2.1 "50% 提前放弃(<10步)" 的真身。但真实机制不是"VLM 消极放弃"，而是 **VLM 非常自信地、主动地判定"已到达"，只是判断错了**。两者需要的修复方向完全不同（"让停止更保守"vs"让目标识别更准"）。

**B 类（40%，锁错物体）高度集中在 image 任务**：

| task_type | B 类中占比 |
| --- | --- |
| image | 29/50 (58%) |
| description | 13/50 (26%) |
| object | 8/50 (16%) |

image 任务是"给参考图找场景里长得像的实例"，一旦第一步选错候选（同场景多个同类实例，如多面 mirror/picture），后续所有"确认到达"都是在自我一致地验证一个从头就错的目标——**根因在目标识别/grounding，不在导航或最后一步对准**。例：`00821-eF36g7L6Z9M_0_3`（mirror, image, 最终 8.05m）、`00813-svBbv1Pavdk_0_0`（plant, object, 7.26m）。

**A 类（29%，锁对但停远）内部也分两种，且过半不是"差一点"**：

| 子类 | n | 特征 |
| --- | --- | --- |
| 近距离（<1.5m） | 20 | 符合"viewpoint 差一点没对准"的直觉，object/image 各 9 个 |
| 远距离尾巴（≥1.5m，最高 12.4m） | 16 | **几乎全是 picture/mirror**，同场景多实例导致 GT id 对应的 3D 位置估计本身就偏了几米，是检测/建图精度问题，不是"最后一步没走近" |

典型例：`00821-eF36g7L6Z9M` 同场景 3 个 picture 子任务（0_2/0_4/0_7）里有 2 个落在 11.5m / 12.4m 的远距离尾巴。

**C 类（31%，从未确认到达）细分**：

| 子类 | n | 解读 |
| --- | --- | --- |
| 步数≥45（耗尽预算） | 30 | 真正的探索超时，与 §2.2 一致 |
| 步数<10即终止 | 5 | 全是 `stair`/`glass` 等结构性/难检测目标，从未生成候选就判定无解 |
| 中途(10-44步) | 4 | 探索方向错误 |

**结论**：旧的"50% 提前放弃 = 停止判定过于激进/悲观"是误读。真正的头部矛盾是**假阳性确认**（A+B, 69%），且 B 类（40%，认错目标）比 A 类（29%，viewpoint 没对准）更大——这与 EXP-002 "让停止更保守"、EXP-004/004b "停止后再走近" 两条已证伪路线的假设都不完全吻合：EXP-002 假设"判定太松需要收紧"，方向对了一半（A+B 确实是假阳性）但用"驳回→重探索"这个动作代价过大（把 A 类近距离样本推远）；EXP-004/004b 假设"锁定的目标是对的，只是 viewpoint 差一点"，但这只覆盖了 A 类的 29%，对 B 类（40%，目标本身就错）无效甚至有害。**修复必须发生在"锁定目标"这一步之前，而不是停止之后。**

## 3. 分场景与分目标物体

### 3.1 场景难度两极分化

| 最难场景 | 成功率 | | 最易场景 | 成功率 |
| --- | --- | --- | --- | --- |
| 00894-HY1NcmCgn3n | **0%** (0/5) | | 00820-mL8ThkuaVTM | 100% (9/9) |
| 00824-Dd4bFSTQ8gi | 14% (1/7) | | 00815-h1zeeAwLh9Z | 100% (5/5) |
| 00800-TEEsavR23oF | 20% (1/5) | | 00878-XB4GS9ShBRE | 90% (9/10) |
| 00808-y9hTuugGdiq | 20% (1/5) | | | |

→ 成功率从 0% 到 100%，**场景结构（大小、布局复杂度）对结果影响极大**。早期只看前几个 scene（如 00820）会严重高估性能——这正是本次"早期 100% → 最终 55%"的原因。

### 3.2 失败高度集中在特定物体类别

| 物体类别 | n | 成功率 | 失败原因推测 |
| --- | --- | --- | --- |
| **大型家具**（fridge/sofa/microwave…） | 81 | **81%** | 体积大、易检测易到达 ✅ |
| 小型装饰（plant/statue/pillow/rug） | 51 | 45% | 体积小、检测难 |
| 平面/反光（mirror/picture/glass） | 80 | **38%** | 反光/平面，检测器与深度都易失效 |
| 结构（stair/handrail） | 11 | **18%** | 非"物体"、无清晰包围盒、viewpoint 难定义 |

**单物体失败率 Top：**
`stair` 100% (6/6)、`glass` 100% (4/4)、`pillow` 100% (3/3)、`statue` 86%、`plant` 73%、`picture` 69%、`mirror` 56%。

→ **两大失败源**：(1) 平面/反光物体（mirror/picture/glass）—— 占失败的一大块，检测器（YOLO-World）和 CLIP 匹配在这类目标上不可靠；(2) 结构性目标（stair/handrail）—— 当前以"物体包围盒 + viewpoint"为核心的成功判定，对楼梯/扶手这类没有紧致 bbox 的目标天然不适配。

## 4. 改进方案（已按 §2.3 的 A/B/C 归因重新校准）

§2.3 的结论改变了优先级：真正的头部矛盾是"锁错目标"（B类,40%）和"锁对但停远"（A类,29%），而不是探索本身；同时 `idea/experiment_log.md` 里已经有三条方向被 278 全集数据**证伪**，不应再投入。

### ❌ 已证伪，不要再做

| 方向 | 实验 | 结果 | 根因 |
| --- | --- | --- | --- |
| 停止后精对准 / refine（走近目标） | EXP-001/004/004b | 3 次全部失败，最差 **−6.4pp** | 没有 GT viewpoint oracle，任何几何 proxy（可见性、距离中心）都不能可靠逼近 GT；"靠近 center"经常是"靠近 GT"的反方向（尤其 image） |
| 停止判定收紧为"驳回→重新探索" | EXP-002 | **−1.80pp**，17 例灾难走失 | 把 0.06–0.24m 的必成功目标判"没到"后放弃重探索，代价（走失到 1–11m）远大于收紧本身的收益 |

→ 结论：**停止之后不要再动 agent**；"收紧确认"本身有效（EXP-002 在 ≤1.0m 口径下反超 baseline），但收紧后的动作不能是"放弃目标"。

### ★★★ 方案 B'：修复"锁错目标"（对应 B 类 40%，image 任务占 58%）—— 最高优先级

同场景同类别多实例（mirror/picture 常见）时，VLM 第一次选中候选就定型，后续"确认到达"只是在自证一个从头错的目标。方向：
- 目标确认前对同类候选做**参考图 vs 候选检测框的相似度排序**（CLIP top-1），而非"第一个过阈值就选"。
- 这正是 Episode Memory 在做的事：EXP-005 的 `exact_id` 精确匹配（58 次命中、0 次回归、净 +6）证明"给 image 任务一个可信锚点"安全有效；EXP-005b 已修复 spatial 匹配的参照系 bug 和 stale-target 归因 bug，**应先跑完 EXP-005b 看 image 任务是否被系统性修复，再考虑独立的 grounding 方案**，避免重复造轮子。

### ★★ 方案 A'：多实例场景的空间去重（对应 A 类远尾 16 个，几乎全是 picture/mirror）

这 16 个"锁对物体但停在 1.5–12.4m 外"的案例，根因是**场景图对同类多实例的 3D 位置估计本身偏了**，不是最后一步没走近（吸取 EXP-004/004b 教训：不要再用停止后移动去补偿建图误差）。应在场景图构建阶段对同类多实例做聚类/去重，让 GT-id 映射到正确的检测簇。

### ★ 方案 C'：结构性目标判定（对应 C 类 <10步 5个：stair/glass）与探索效率（C 类超时 30个）

- 结构性目标（stair/handrail）无紧致 bbox，"到达 viewpoint"定义模糊 → 改用"可视即成功/更宽到达半径"。
- 超时 30 个（平均离目标 7.3m）沿用原思路：frontier 选择引入房间语义先验（M3DSG 已有 room label），减少盲目横跳。

### 工程（不紧急，可并行）

- 跑全集（`--splits 10`）对齐论文 Table 1（当前 278 首集只对齐消融 Table 3）。
- 修复 `records_*.jsonl` 重复写入（每 subtask 写 ~3 次，不影响指标，只影响文件体积）。
- 换 Qwen-VL-Max 做同口径对比（论文主力配置）。

## 5. 建议的下一步（优先级排序）

1. **【等结果】** EXP-005b（Episode Memory 修复版）跑完后做因果分解，重点看 image 任务 + `spatial` match 是否不再回归——这是当前唯一已在验证中的、直接命中 B 类根因（锁错目标）的方向。
2. **【新方向，中成本】** 若 EXP-005b 对 image 任务提升有限，再实现方案 B'（候选目标 CLIP 相似度排序）单独验证。
3. **【小而确定】** 方案 C' 的结构性目标判定放宽（stair/glass 仅 9 个样本，改动小、无风险）。
4. **【长尾，低优先级】** 方案 A'（多实例场景图去重）、探索语义先验、全集评测/Qwen 对比。
5. **不要做**：任何"停止后再移动 agent"的 refine 类改动，或"判定失败就放弃目标重探索"的 reject 类改动——已用 5 次独立实验证伪。
